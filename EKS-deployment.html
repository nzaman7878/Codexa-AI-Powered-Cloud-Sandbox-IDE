What is EKS?
Key concept before you start
Core Concept

EKS (Elastic Kubernetes Service) is AWS's managed Kubernetes service. AWS runs and maintains the control plane (the master node that manages the whole cluster) for you. You only manage the worker nodes — the EC2 machines where your pods actually run.

Local vs EKS

What Local (Docker Desktop) EKS (AWS)
Control plane You manage everything AWS manages it for you
Docker images Stored locally on your machine Must be pushed to ECR (AWS registry)
Metrics server Pre-installed You install it manually
Autoscaling Works out of box Requires setup (HPA + Cluster Autoscaler)
Cost Free ~$0.10/hr control plane + ~$0.04/hr per node
Domain / SSL localhost / none Real domain + free SSL via ACM
Frontend hosting Vite dev server S3 + CloudFront (static CDN)
01
Full Architecture
Frontend + backend on a single domain with a clear split
The Architecture

Your entire app lives on one domain (e.g. www.cryboy.in). CloudFront sits at the edge and decides where to send each request: anything matching /api/* goes to the EKS Load Balancer, everything else is served as a static file from S3. Backend microservices live inside EKS and are routed by an nginx ingress controller.

Traffic flow

Browser hits www.cryboy.in
DNS CNAME resolves to your CloudFront distribution
CloudFront splits the request
Path-based behaviors decide which origin handles the request: S3 for static assets, EKS Load Balancer for /api/*
Path = / → S3 (React build)
Static index.html, JS, and CSS files served directly from S3
Path = /api/* → EKS Load Balancer
Forwarded to AWS Load Balancer created by the nginx ingress controller
nginx Ingress reads rules
Routes /api/auth → auth-service, /api/ai → ai-service, /api/sandbox → sandbox-service, etc.
Service load-balances to pods
Kubernetes Service finds pods via label selector and distributes traffic across replicas

IAM User & AWS CLI Setup
Create the IAM user with the right policies, then configure CLI
Key Concept — Why IAM matters

AWS uses IAM (Identity and Access Management) to control what each user is allowed to do. When you run eksctl, aws ecr, aws s3, or aws cloudfront, AWS checks whether your IAM user has permission. Without the right policies attached, commands fail with AccessDenied. This is the #1 source of "why is this failing" pain in AWS deployments.

Step 1 — Create a dedicated IAM user

Never use your AWS root account for CLI operations. Create a separate IAM user just for deployments.

Go to IAM → Users → Create user
AWS Console → search "IAM" → Users (left sidebar) → Create user
Name the user
Something descriptive like eks-deploy-user. Do NOT tick "Provide user access to the AWS Management Console" — this user only needs CLI access.
Attach policies directly
On the permissions step choose "Attach policies directly". Attach each policy from the table below.
Create user → Create access key
After the user is created → Security credentials tab → Create access key → "Command Line Interface (CLI)". Copy both keys immediately — the secret key is shown only once.

Step 2 — Required IAM policies

Attach every policy listed below. Each one covers a different AWS service this deployment touches. Missing any single one will cause commands to fail mid-way.

Policy Name What it allows Why you need it
AmazonEKSFullAccess All EKS API actions including DescribeClusterVersions, CreateCluster, DescribeCluster Required for every eksctl command. Without this, even eksctl create cluster fails immediately.
AmazonEKSClusterPolicy Lets the cluster manage AWS resources on your behalf Required for cluster control-plane operations.
AmazonEKSWorkerNodePolicy Lets EC2 nodes join the EKS cluster and receive workloads Without this, worker nodes cannot register with the control plane.
AmazonEC2ContainerRegistryFullAccess Create repositories, push and pull images in ECR Required for aws ecr create-repository, docker push, and EKS pulling images.
AmazonVPCFullAccess Create and manage VPCs, subnets, security groups, route tables eksctl creates a full VPC for your cluster. Without this it fails mid-creation.
IAMFullAccess Create and attach IAM roles and policies eksctl creates IAM roles for node groups automatically. Also needed for cluster autoscaler.
AmazonEC2FullAccess Launch, manage, and terminate EC2 instances and related resources EKS worker nodes are EC2 instances. eksctl needs this to provision and destroy them.
CloudFormationFullAccess Create and manage CloudFormation stacks eksctl uses CloudFormation stacks internally for every resource. Mandatory for any eksctl command.
AmazonS3FullAccess Create buckets, upload files, set bucket policies, configure static website hosting Required for hosting the React frontend on S3 — bucket creation, file sync, and public-access configuration.
CloudFrontFullAccess Create and update CloudFront distributions, behaviors, origins, invalidations Required for serving the React frontend via CloudFront CDN with the API routing behavior.
AWSCertificateManagerFullAccess Request, validate, and manage SSL/TLS certificates via ACM Required for the SSL certificate that CloudFront uses for your custom domain.
💡

If you want a shortcut for a personal learning account: attach AdministratorAccess — it covers all the above. Never do this in a production or shared account — always use the minimum permissions principle.

⚠️

Never share or commit your Access Key ID or Secret Access Key. Do not paste them in code, push them to GitHub, or send them in chat. If a key leaks, go to IAM → Users → your user → Security credentials → Deactivate the key immediately, then create a new one.

Create EKS Cluster
Spin up worker nodes on AWS — takes 15–20 minutes
Terminal 
eksctl create cluster \
  --name express-cluster \
  --region ap-southeast-1 \
  --nodegroup-name express-nodes \
  --node-type t3.medium \
  --nodes 2 \
  --nodes-min 1 \
  --nodes-max 3 \
  --managed
Flag Value What it does
--name express-cluster Cluster name. Used in all future eksctl commands.
--region ap-southeast-1 AWS region where the cluster is created.
--nodegroup-name express-nodes Name of the group of worker nodes.
--node-type t3.medium EC2 instance size: 2 vCPU, 4 GB RAM. ~$0.04/hr per node.
--nodes 2 Starting number of worker nodes.
--nodes-min 1 Minimum nodes Cluster Autoscaler keeps running.
--nodes-max 3 Maximum nodes Cluster Autoscaler can scale up to.
--managed (no value) Makes AWS manage node OS patches and updates automatically.
⚠️

Do not close the terminal while this runs. It takes 15–20 minutes and creates VPC, subnets, IAM roles, and EC2 instances behind the scenes.

Common IAM errors during cluster creation

Error Cause Fix
eks:DescribeClusterVersions AccessDenied Missing AmazonEKSFullAccess policy Go to IAM → your user → Add permissions → attach AmazonEKSFullAccess
cloudformation:CreateStack AccessDenied Missing CloudFormation policy Attach CloudFormationFullAccess
ec2:CreateVpc AccessDenied Missing VPC/EC2 permissions Attach AmazonVPCFullAccess and AmazonEC2FullAccess

Batch ECR Setup
Create one ECR repository per microservice — all in a single command
Key Concept

ECR (Elastic Container Registry) is AWS's private Docker image registry. Every microservice needs its own ECR repository. With a multi-service app you do not want to create them one by one — use a loop instead.

Step 1 — Create all ECR repositories in one shot

Replace the service list with your own service names. This creates a repo for each in a single command:

Terminal one-liner
Terminal 
for svc in ai-orchestration auth notification agent router sandbox template; do
  aws ecr create-repository --repository-name $svc --region ap-southeast-1
done

Alternative — reusable shell script

If your team needs this regularly, save it as a script:

create-ecr-repos.sh 
#!/bin/bash

REGION="ap-southeast-1"
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)

# List every microservice here
SERVICES=("ai-orchestration" "auth" "notification" "agent" "router" "sandbox" "template")

for SERVICE in "${SERVICES[@]}"; do
  echo "Creating ECR repo: $SERVICE"
  aws ecr create-repository \
    --repository-name $SERVICE \
    --region $REGION
  echo "✅ Done: $ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/$SERVICE"
done
Terminal — run the script 
chmod +x create-ecr-repos.sh
./create-ecr-repos.sh

Step 2 — Login Docker to ECR

You only need to do this once per terminal session (the token lasts 12 hours):

Terminal 
aws ecr get-login-password --region ap-southeast-1 | \
  docker login --username AWS \
  --password-stdin <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com
Verify all repos were created
Terminal 
aws ecr describe-repositories --region ap-southeast-1 \
  --query "repositories[].repositoryName"

Skaffold Deploy
Build, push, and deploy all microservices with one command
Key Concept — Why Skaffold for multi-service

Skaffold reads a single config file listing every service in your project. With one command it builds every Docker image with the correct platform, pushes them all to their matching ECR repos with content-based digest tags, patches your Kubernetes manifests with the new digests, and applies them to the cluster. For a 7-service app, this replaces 28 manual commands with one.

Local dev vs EKS — keep them separate

You already have a skaffold.yaml for local development (uses local images, no ECR push). Create a second file specifically for EKS that prefixes every image with the ECR URI and forces linux/amd64 builds. This way you can switch between local dev and EKS deploys without editing files.

Step 1 — Create skaffold-eks.yaml in your project root

skaffold-eks.yaml
skaffold-eks.yaml 
apiVersion: skaffold/v4beta13
kind: Config

build:
  tagPolicy:
    sha256: {}

  platforms:
    - linux/amd64        # EKS t3.medium nodes are x86_64 — Mac M1/M2/M3 must cross-build

  artifacts:

    - image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/ai-orchestration
      context: ai-orchestration
      docker:
        dockerfile: dockerfile

    - image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/auth
      context: auth
      docker:
        dockerfile: dockerfile

    - image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/notification
      context: notification
      docker:
        dockerfile: dockerfile

    - image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/agent
      context: sandbox/agent
      docker:
        dockerfile: dockerfile

    - image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/router
      context: sandbox/router
      docker:
        dockerfile: dockerfile

    - image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/sandbox
      context: sandbox/server
      docker:
        dockerfile: dockerfile

    - image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/template
      context: sandbox/template
      docker:
        dockerfile: dockerfile

manifests:
  rawYaml:
    - k8s/ai-deployment.yml
    - k8s/ai-service.yml
    - k8s/auth-deployment.yml
    - k8s/auth-service.yml
    - k8s/sandbox-deployment.yml
    - k8s/sandbox-service.yml
    - k8s/router-deployment.yml
    - k8s/router-service.yml
    - k8s/notification-deployment.yml
    - k8s/notification-service.yml
    - k8s/ingress.yml
    - k8s/rbac.yml
    - k8s/secrets.yml
Key Value What it does
tagPolicy.sha256 {} Tags images with their content digest (sha256). Guarantees pods always pull the exact new version.
platforms linux/amd64 Forces amd64 build on Apple Silicon. Without this, pods crash with "exec format error" on EKS.
artifacts[].image Full ECR URI (no tag) Skaffold appends the digest. Each artifact maps to one ECR repo.
artifacts[].context folder path Folder containing this service's Dockerfile.
manifests.rawYaml list of YAML files Skaffold applies all these manifests after building. It patches the image fields with the new digests automatically.

Step 2 — Update every deployment.yaml to reference the ECR URI

Each microservice's deployment manifest must reference the same image URI as in skaffold-eks.yaml. Critical: the image must match exactly (without tag). If it does not match, Skaffold will not patch it and your pod will fail with can't be pulled.

k8s/auth-deployment.yml — example 
containers:
  - name: auth-container
    # ❌ Wrong — local name. Will fail with "auth:latest can't be pulled"
    image: auth:latest

    # ✅ Correct — matches the artifact in skaffold-eks.yaml exactly (no tag)
    image: <ACCOUNT_ID>.dkr.ecr.ap-southeast-1.amazonaws.com/auth
    imagePullPolicy: Always
⚠️

Check every single deployment.yaml. If one service still has a local image name, only that pod will fail — the rest deploy fine, but you will see the error <service-name>:latest can't be pulled. Run grep -r "image:" k8s/ to audit all files at once.

Step 3 — Run Skaffold

Deploy everything once
Terminal 
skaffold run -f skaffold-eks.yaml

Builds every image with linux/amd64, pushes each to its ECR repo with a sha256 digest tag, patches the manifests with the new digests, and applies everything. Resources stay live after the command exits.

Important — use `run` not `dev` for EKS

skaffold dev watches files and tears down everything on Ctrl+C. skaffold run deploys once and exits, leaving your app running. Always use run for EKS — dev is for local Docker Desktop only.

Verify all pods are running
Terminal 
kubectl get pods

# Every pod should show: 1/1 Running
NAME                                       READY   STATUS    AGE
ai-deployment-645686bfbb-5dh2x             1/1     Running   8m
auth-deployment-6c464c5479-rj6zc           1/1     Running   8m
notification-deployment-c9c4c4dcc-h8zrk    1/1     Running   8m
router-deployment-59dc69ccfd-jfgwb         1/1     Running   8m
sandbox-deployment-6766879bcc-m69bg        1/1     Running   8m

Common Skaffold errors

Error Cause Fix
no basic auth credentials Docker not logged in to ECR Re-run aws ecr get-login-password | docker login. Token expires every 12h.
exec format error Image built for arm64 instead of amd64 Add platforms: [linux/amd64] to the build section of skaffold-eks.yaml
<service>:latest can't be pulled deployment.yaml still has local image name Update that deployment to use the full ECR URI matching skaffold-eks.yaml exactly
manifest unknown ECR repository does not exist Re-run the batch ECR creation script for the missing service

nginx Ingress Controller
The pod that receives external traffic and creates an AWS Load Balancer
Key Concept

Your ingress.yml is just a set of rules — which path goes to which service. It does nothing without an Ingress Controller, the actual nginx pod that reads those rules and routes real traffic. On EKS, installing the controller also automatically provisions an AWS Network Load Balancer, giving you a public hostname.

Install the ingress controller
Terminal 
kubectl apply -f \
  https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.12.1/deploy/static/provider/cloud/deploy.yaml
Get the Load Balancer URL
Terminal 
kubectl get service -n ingress-nginx

# Look for EXTERNAL-IP — this is your Load Balancer hostname:
NAME                       TYPE           EXTERNAL-IP
ingress-nginx-controller   LoadBalancer   a8eacd25575f94fbb9b793f1a89a9d24-532281847.ap-southeast-1.elb.amazonaws.com

Copy this URL. You will use it as an origin in CloudFront (Section 12) and optionally as a CNAME for wildcard subdomains.

Example ingress.yml for multi-service routing

This is the actual ingress used in this deployment — routes /api/* paths to backend services, plus wildcard subdomains for the router service:

k8s/ingress.yml 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: codespace-ingress
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "6000"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "6000"
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "6000"
    nginx.ingress.kubernetes.io/affinity: "cookie"
    nginx.ingress.kubernetes.io/session-cookie-name: "route"
spec:
  ingressClassName: nginx
  rules:
  - http:
      paths:
      - pathType: Prefix
        path: "/api/auth"
        backend:
          service:
            name: auth-service
            port:
              number: 80
      - pathType: Prefix
        path: "/api/ai"
        backend:
          service:
            name: ai-service
            port:
              number: 80
      - pathType: Prefix
        path: "/api/sandbox"
        backend:
          service:
            name: sandbox-service
            port:
              number: 80

  - host: "*.preview.cryboy.in"
    http:
      paths:
      - pathType: Prefix
        path: "/"
        backend:
          service:
            name: router-service
            port:
              number: 80
💡

-n ingress-nginx means "look in the ingress-nginx namespace". The ingress controller installs in its own namespace, not the default one where your pods live.

ACM SSL Certificate
Request before CloudFront — order matters
Critical — Order of operations

CloudFront requires an SSL certificate to use a custom domain like www.cryboy.in. The certificate must be validated and ISSUED before you create the CloudFront distribution — if it is still pending, CloudFront rejects it with InvalidViewerCertificate. Always do ACM first, wait until it shows ISSUED, then create CloudFront.

⚠️

ACM certificates for CloudFront MUST be in us-east-1. CloudFront only accepts certificates from us-east-1 (N. Virginia) regardless of where your other resources are. If you create the cert in any other region, CloudFront cannot use it.

Step 1 — Request the certificate

Terminal 
aws acm request-certificate \
  --domain-name cryboy.in \
  --subject-alternative-names "*.cryboy.in" \
  --validation-method DNS \
  --region us-east-1

Returns a CertificateArn like:

Response 
{
    "CertificateArn": "arn:aws:acm:us-east-1:329599656829:certificate/065ee0d2-887e-460f-b6eb-6c10684e4895"
}

Save this ARN — you will pass it to CloudFront in the next section.

Step 2 — Get the DNS validation record

Terminal 
aws acm describe-certificate \
  --certificate-arn <YOUR_CERT_ARN> \
  --region us-east-1 \
  --query "Certificate.DomainValidationOptions"

Returns the CNAME you need to add to DNS:

Response 
{
  "ResourceRecord": {
    "Name": "_18fdf25abc2491e0e5cdf8b20ac9578a.cryboy.in.",
    "Type": "CNAME",
    "Value": "_6b94eb7f375219e602eeadb0371334fe.jkddzztszm.acm-validations.aws."
  }
}

Step 3 — Add the CNAME in your DNS provider

Open your DNS provider's dashboard and add a CNAME record with:

Field Value Note
Type CNAME —
Name / Host _18fdf25abc2491e0e5cdf8b20ac9578a Strip the domain suffix. ACM gives you _xxx.cryboy.in. but most DNS providers append the domain automatically — paste only the subdomain part.
Data / Value _6b94eb7f375219e602eeadb0371334fe.jkddzztszm.acm-validations.aws Remove the trailing dot if your provider rejects it.
⚠️

Common mistake: Pasting the Name as _xxx.cryboy.in instead of _xxx. The DNS provider then stores it as _xxx.cryboy.in.cryboy.in and ACM cannot find it. If validation stays pending for over 15 minutes, this is almost always the reason.

Step 4 — Wait for ISSUED status

ACM polls your DNS every few minutes. Run this every 2-3 minutes until it returns "ISSUED":

Terminal 
aws acm describe-certificate \
  --certificate-arn <YOUR_CERT_ARN> \
  --region us-east-1 \
  --query "Certificate.Status"

Typically takes 5–15 minutes after the DNS record is correctly set. Once it returns "ISSUED" you can move to CloudFront. Until then, every CloudFront create-distribution call will fail with InvalidViewerCertificate.

S3 Bucket for React Frontend
Build React, create a public bucket, sync the dist folder

Step 1 — Configure React for production API calls

In dev, Vite proxies /api/* to your local backend. In production CloudFront will route /api/* for you, so your React code can keep using relative paths like /api/auth/login — no code changes needed. The build output will work as-is.

💡

If your React code uses absolute URLs (e.g. http://localhost:3000/api/...), replace them with relative paths (/api/...) or use an env variable like VITE_API_BASE_URL. Relative paths are simpler — CloudFront handles the routing.

Step 2 — Build the React app

Terminal 
cd frontend
npm run build
# Vite outputs static files to ./dist

Step 3 — Create the S3 bucket

Terminal 
aws s3 mb s3://cryboy-frontend --region ap-southeast-1

Step 4 — Disable Block Public Access

S3 buckets are private by default. CloudFront needs to read the files, so disable the block:

Terminal 
aws s3api put-public-access-block \
  --bucket cryboy-frontend \
  --public-access-block-configuration \
  "BlockPublicAcls=false,IgnorePublicAcls=false,BlockPublicPolicy=false,RestrictPublicBuckets=false"

Step 5 — Enable static website hosting

Terminal 
aws s3 website s3://cryboy-frontend \
  --index-document index.html \
  --error-document index.html
💡

Set the error document to index.html too. This is critical for React Router. Without it, direct URLs like cryboy.in/dashboard return a 403 instead of loading your SPA. The same fix is also done at the CloudFront layer in Section 11.

Step 6 — Set bucket policy for public read

Terminal 
aws s3api put-bucket-policy \
  --bucket cryboy-frontend \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [{
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::cryboy-frontend/*"
    }]
  }'

Step 7 — Upload dist/ to S3

Terminal 
aws s3 sync dist/ s3://cryboy-frontend --delete
Flag What it does
sync Uploads only changed files — faster than cp --recursive.
--delete Removes files from S3 that no longer exist in dist/. Essential — otherwise old chunks accumulate and serve broken JS.
Verify upload
Terminal 
aws s3 ls s3://cryboy-frontend
# Should list index.html, assets/, etc.

curl -I http://cryboy-frontend.s3-website-ap-southeast-1.amazonaws.com
# Should return 200 — confirms the S3 website endpoint works directly
11
CloudFront Distribution
Serve the React app over HTTPS with your custom domain
⚠️

Before creating CloudFront, verify your ACM certificate is ISSUED. If it is still PENDING_VALIDATION, CloudFront will reject the request with InvalidViewerCertificate.

Step 1 — Create the CloudFront distribution

Terminal 
aws cloudfront create-distribution \

Step 2 — Add custom error responses for React Router

React Router uses client-side routing — direct URLs like www.cryboy.in/dashboard are not real S3 files. Without this fix, CloudFront returns a 403/404 for any non-root URL. The fix is to serve /index.html with status 200 for both error codes, so React Router can handle the route on page load.

Do this in the AWS Console — it's clearer than the JSON:

Open CloudFront Console → your distribution
Find the distribution by ID and click in
Click Error pages tab → Create custom error response
Two records to create — one for 403 and one for 404
For each: code → /index.html → 200
HTTP error code: 403, Response page path: /index.html, HTTP response code: 200. Save. Then repeat for 404.

API Routing via CloudFront
Single-domain split: /api/* → EKS, everything else → S3
Key Concept — Single domain, two origins

To serve frontend and backend from the same domain, CloudFront uses path-based routing. You add the EKS Load Balancer as a second origin, then create a behavior that routes /api/* requests to EKS while the default behavior keeps serving S3 for everything else. Your React code can use relative /api/... URLs and they just work.

Do this in the AWS Console — it's faster and less error-prone than CLI

Step 1 — Add EKS Load Balancer as a second origin

CloudFront → your distribution → Origins tab → Create origin
—
Origin domain
Paste your EKS Load Balancer URL (from kubectl get svc -n ingress-nginx)
Protocol: HTTP only · Port: 80
nginx ingress serves plain HTTP. HTTPS is handled at the CloudFront edge.
Name: EKS-backend → Save
Distinct name from the S3 origin so you can target it in a behavior

Step 2 — Create a behavior for /api/*

Field Value Why
Path pattern /api/* Matches any URL starting with /api/. The default behavior (S3) handles everything else.
Origin EKS-backend Route matched requests to your EKS Load Balancer.
Viewer protocol policy Redirect HTTP to HTTPS Force HTTPS for all API calls.
Allowed HTTP methods GET, HEAD, OPTIONS, PUT, POST, PATCH, DELETE APIs need every method, not just GET. Default behavior allows only GET, HEAD — that breaks POST requests.
Cache policy CachingDisabled Never cache API responses — they are dynamic and per-user. CloudFront has a built-in policy named exactly "CachingDisabled".
Origin request policy AllViewer Forwards all viewer headers, cookies, and query strings to your backend. Required for auth cookies and request bodies.
⚠️

If you forget to enable POST/PUT/DELETE on the behavior, your API will appear to work for GET requests but break on login, sign-up, or any form submission. Symptoms: 405 Method Not Allowed or requests silently returning the S3 HTML instead of the API JSON.

Step 3 — Wait for the distribution to redeploy

Any change to CloudFront triggers a global redeploy. Status moves from Deployed → InProgress → Deployed again, takes 10–15 minutes.

Terminal 
aws cloudfront get-distribution \
  --id <DISTRIBUTION_ID> \
  --query "Distribution.Status" \
  --output text
# Run every 2-3 minutes until it returns "Deployed"
Verify both origins respond correctly
Terminal 
curl -I https://www.cryboy.in
# Should return 200 from CloudFront with server: AmazonS3 → React app is served

curl -I https://www.cryboy.in/api/auth
# Should hit the auth service — look for X-Powered-By: Express in the response
# If you see server: AmazonS3 instead, the /api/* behavior is missing or wrong

# Should return the EKS Load Balancer hostname/IP
⏱️

DNS changes take 5–30 minutes to propagate globally, sometimes longer depending on your TTL. If your provider has a 4-hour TTL on the old record, the cached version may stick around. Set TTL to something low (300s) when first setting up so iteration is fast.

Autoscaling on EKS
Two levels — pods and nodes — both need setup
Key Concept — Two types of autoscaling

HPA (Horizontal Pod Autoscaler) adds or removes pods based on CPU. Cluster Autoscaler adds or removes EC2 nodes when pods cannot be scheduled. Both are disabled by default on EKS because AWS does not know your scaling limits or budget. You install and configure both yourself.

Level 1 — Pod Autoscaling (HPA)

HPA needs a metrics server to read CPU usage. EKS does not have one pre-installed.

Install Metrics Server
Terminal 
kubectl apply -f \
  https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
Create HPA for a service
Terminal 
kubectl autoscale deployment auth-deployment \
  --min=1 \
  --max=5 \
  --cpu-percent=50

Level 2 — Node Autoscaling (Cluster Autoscaler)

Without this, HPA adds pods but they get stuck Pending when nodes are full. Cluster Autoscaler adds new EC2 nodes automatically.

Step 1 — Enable OIDC
Terminal 
eksctl utils associate-iam-oidc-provider \
  --cluster express-cluster \
  --region ap-southeast-1 \
  --approve
Step 2 — Create IAM service account
Terminal 
eksctl create iamserviceaccount \
  --cluster express-cluster \
  --namespace kube-system \
  --name cluster-autoscaler \
  --attach-policy-arn arn:aws:iam::aws:policy/AutoScalingFullAccess \
  --approve
Step 3 — Install Cluster Autoscaler
Terminal 
kubectl apply -f \
  https://raw.githubusercontent.com/kubernetes/autoscaler/master/cluster-autoscaler/cloudprovider/aws/examples/cluster-autoscaler-autodiscover.yaml

kubectl -n kube-system set env deployment/cluster-autoscaler \
  CLUSTER_NAME=express-cluster

Full autoscaling picture

Traffic spike hits
    ↓
CPU per pod exceeds 50%
    ↓
HPA adds more pods
    ↓
Nodes run out of space
    ↓
Cluster Autoscaler adds new EC2 node
    ↓
Pending pods get scheduled ✅

Updating Services
Roll out backend code changes and React rebuilds

Update one or more microservices

Skaffold detects which contexts have changed and rebuilds only those. Just re-run the same command:

Terminal 
skaffold run -f skaffold-eks.yaml

Each changed service gets a new sha256 digest, gets pushed to ECR, and the deployment is patched. Kubernetes rolls out new pods with zero downtime — old pods stay alive until the new ones are Ready.

Rollback a bad deploy

Terminal 
kubectl rollout undo deployment auth-deployment
# Rolls back to the previous revision instantly

kubectl rollout history deployment auth-deployment
# Lists all revisions so you can roll back to a specific one

Update React frontend

Terminal 
cd frontend
npm run build
aws s3 sync dist/ s3://cryboy-frontend --delete

# Invalidate CloudFront cache so users immediately get the new build
aws cloudfront create-invalidation \
  --distribution-id <DISTRIBUTION_ID> \
  --paths "/*"
💡

Always invalidate after a frontend deploy. CloudFront caches your old index.html for hours — without invalidation users keep seeing the old version even after the new files are in S3. The /* path clears the whole cache. Each CloudFront account gets 1,000 free invalidations per month.

Cleanup
Stop AWS charges when you are done
⚠️

EKS worker nodes cost ~$0.04/hr per node plus ~$0.10/hr for the control plane. CloudFront and S3 are cheap but not free. Always tear down resources when you stop using them.

Delete the entire EKS cluster
Terminal 
eksctl delete cluster \
  --name express-cluster \
  --region ap-southeast-1

Deletes the cluster, all EC2 nodes, VPC, subnets, IAM roles — everything eksctl created.

Delete CloudFront + S3
Terminal 
# 1. Disable CloudFront distribution first (takes ~15 mins)
#    Console: CloudFront → distribution → Disable → wait for "Deployed" → Delete

# 2. Then empty and delete S3 bucket
aws s3 rm s3://cryboy-frontend --recursive
aws s3 rb s3://cryboy-frontend
Delete ECR repositories
Terminal 
for svc in ai-orchestration auth notification agent router sandbox template; do
  aws ecr delete-repository --repository-name $svc --region ap-southeast-1 --force
done
Remove only the app (keep cluster)
Terminal 
skaffold delete -f skaffold-eks.yaml
# Removes everything skaffold-eks.yaml applied — Deployments, Services, Ingress
